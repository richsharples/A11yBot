A11yBot · Design review · May 2026

One product,
two design languages.

A11yBot ships with a strong brand idea — a phosphor-green CRT terminal — but the idea only lives in the chrome. The moment a PM enters the actual work surface, they cross into a generic light SaaS UI that doesn't know the brand exists. This review names what's drifted, then proposes a single system that lets the terminal identity carry through the whole product without sacrificing professionalism for a VPAT-grade tool.

01
The short version

Three things to fix, in priority order.

01

Unify the two visual zones.

The terminal chrome and the light working surface should look like deliberate halves of one instrument — not two screens stapled together. Today the seams are accidents, not design decisions.

02

Replace the candy palette with semantic tokens.

Six unrelated Tailwind color families currently overlap in meaning. Replace them with four calibrated semantic tones (issue, info, warn, ok) and stop using phosphor-green for both brand and success.

03

Earn the right to make an accessibility tool.

The header text at #39FF14 @ 30–50% opacity on #0b1a0d fails WCAG AA. A tool that audits VPATs cannot ship UI that wouldn't pass its own scan. This is the single most embarrassing risk.

The fix isn't to abandon the phosphor look. It's to (a) use full-strength phosphor for any text that needs to be readable, (b) push text below comfortable contrast up to ≥4.5:1 by lifting alpha, lightening to a soft phosphor tint like #9af48a, or both, and (c) raise minimum text size to 12px and minimum tap target to 32px. The CRT mood survives intact — it just stops failing its own audit.

05
The proposal

One product. Two zones, on purpose.

The brand identity is genuinely good — the CRT phosphor robot is memorable and unique in a procurement-tool category that mostly looks like 2014 Bootstrap. The fix isn't to throw it away; it's to commit to it, then give it a calm working surface to operate against.

The instrument zone

CRT phosphor — header, status bar, brand moments, and any "tool that's working" surface (scan progress, log tail, model status). Mono type. Dark CRT background. Phosphor accent at full strength. This is where the bot lives.

The working surface

Warm paper — sidebar, criteria list, detail panel, setup form. Plex Sans body, Plex Mono only for identifiers (criterion refs, file paths, rule IDs). Muted semantic palette. This is where the PM lives and reads dense compliance language for hours.

Principles

  1. Phosphor green is a brand color, not a success color. It only appears on the dark instrument zone. Saving things is dark-ink-on-paper, not green.
  2. Mono is for identifiers and the instrument. Sans is for content. Don't mix within a paragraph.
  3. Semantic color is muted and warm — terra-cotta, navy, ochre, forest — so a screen full of "issues" doesn't read as a panic. PMs need to live here.
  4. One primary action per screen. Today every button looks primary. Tomorrow only the next required step does.
  5. The instrument acknowledges activity, the surface acknowledges progress. Status log lives in the dark bar. Per-criterion confidence and "PM confirmed" indicators live on paper.

Fig. 1 — Review screen, redesigned. Same data, same actions, one design language.

What changed at the top

Header gets a tab spine (review · evidence · settings) and a single phosphor primary CTA (Create report). Metadata is labeled, not pipe-separated. Progress moves to the right with prev/next as a single bordered cluster.

What changed in the middle

Sidebar gets chapter numbers and a clearer hierarchy of name/sub/badges. Criterion list leads with the conformance chip — so it reads like a status board at a glance. The detail panel reads top-down: definition → evidence → conformance → action row, separated by mono-eyebrow rules.

What changed at the bottom

Status bar keeps the CRT mood but pushes contrast up to comfortable. Empty/idle states use phosphor-soft, not 30%-alpha phosphor. The collapse caret moves to the left where the user expects it.

09
If you can only do five things

An order of operations.

1

Replace globals.css tokens.

Throw out the shadcn HSL palette and define the 24 named tokens from §06 (paper, ink, crt, phosphor, semantic). Update tailwind.config.ts to expose them. Half a day. Everything downstream gets cheaper.

2

Fix the contrast failures in the header and status bar.

Replace text-[#39FF14]/30 and /50 with phosphor-soft (#9af48a) or full phosphor on dark. Promote all 10px labels to 11.5px minimum. This alone removes the most visible accessibility risk in the product.

3

Consolidate the button vocabulary.

Build four React button variants: Button.Primary, Button.Secondary, Button.Ghost, Button.Danger, plus an InstrumentButton for dark zones. Remove every inline button class. Use ink-on-paper for save / confirm — drop the green button forever.

4

Refactor evidence and chips into three components.

<Chip level> for the five conformance levels, <Badge variant> for issue / info / warn / ok / neutral, <Banner variant> for the issue and warn callouts in CriterionDetail. Replace the yellow AI-inferred banner, the orange issue banner, and the inline source/app counters with these three components.

5

Quiet the modal.

The "Discard & start new" modal currently inherits the full phosphor terminal aesthetic and feels like a different app interrupting. Move it onto paper, give it a sober ink title and a text-only danger button — destructive confirmation should feel grave, not glowing.

What to leave alone. The information architecture (3-pane review, chapter → criterion → detail), the workflow (interview · AI draft · confirm · export), the keyboard model (Space = confirm & next), and the auto-scan-on-create behavior all work. Don't touch them. This is a coat of paint, not a rebuild — the bones are good.
